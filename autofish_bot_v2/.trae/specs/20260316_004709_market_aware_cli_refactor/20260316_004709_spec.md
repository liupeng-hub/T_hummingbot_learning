# Market Aware Backtest CLI 参数重构规范

## 背景

当前 `market_aware_backtest.py` 的 CLI 参数过于分散和随意（约 20+ 个独立参数），导致：
1. 外部调用难以适配
2. 参数组合复杂，难以维护
3. 测试计划配置不清晰

需要将参数分组整理为结构化的 JSON 参数，提高可维护性和易用性。

## 变更内容

- 重构 `market_aware_backtest.py` 的 CLI 参数结构（**破坏性变更**：移除旧参数）
- 改造 `test_plans` 表结构，增加专用字段
- 更新 `test_manager` Web 页面支持新参数配置

## 影响范围

- Affected code: `market_aware_backtest.py`, `database/test_results_db.py`, `test_manager.py`
- Affected tables: `test_plans`
- **破坏性变更**: 现有 CLI 调用方式需要更新，旧参数全部移除

## 新增需求

### 需求：CLI 参数重构

`market_aware_backtest.py` 的 CLI 参数应按以下结构组织：

#### 新参数结构

```
必选参数:
  --symbol          标的（如 BTCUSDT）
  --date-range      时间范围（如 20200101-20260310）

可选参数:
  --amplitude-params        振幅参数（JSON字符串）
  --market-algorithm-params 行情算法参数（JSON字符串）
  --entry-algorithm-params  入场算法参数（JSON字符串）
  --timeout-first-entry-params  首次入场超时参数（JSON字符串）
```

**说明**:
- 输出目录自动生成（out/test_report/market_aware/plan_id_时间戳）
- 测试结果始终保存到数据库

#### 场景：振幅参数默认值
- **当** 用户不传入 `--amplitude-params`
- **则** 从配置文件中读取 decay_factor=0.5 对应的振幅配置 JSON
- **说明**: 配置文件中包含不同 decay_factor 对应的完整振幅配置

#### 场景：行情算法参数默认值
- **当** 用户不传入 `--market-algorithm-params`
- **则** 使用"总是震荡"（always_ranging）方式
- **默认值**:
  ```json
  {
    "algorithm": "always_ranging"
  }
  ```
- **说明**: always_ranging 算法没有其他参数

#### 场景：入场算法参数默认值
- **当** 用户不传入 `--entry-algorithm-params`
- **则** 不使用入场算法（使用配置文件中的入场价格策略）
- **默认值**: `{}` (空对象，表示不使用)

#### 场景：首次入场超时参数默认值
- **当** 用户不传入 `--timeout-first-entry-params`
- **则** a1_timeout_minutes 为 0（表示不启用首次入场超时）
- **默认值**:
  ```json
  {
    "a1_timeout_minutes": 0
  }
  ```
- **参数说明**:
  - `a1_timeout_minutes`: 超时时间（分钟），0 表示不启用

### 需求：数据库表结构改造

`test_plans` 表应支持 market_aware 测试类型的专用字段：

#### 新增字段

| 字段名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| symbol | TEXT | - | 标的 |
| date_range | TEXT | - | 时间范围 |
| amplitude_params | TEXT | '{}' | 振幅参数（JSON） |
| market_algorithm_params | TEXT | '{}' | 行情算法参数（JSON） |
| entry_algorithm_params | TEXT | '{}' | 入场算法参数（JSON） |
| timeout_first_entry_params | TEXT | '{}' | 首次入场超时参数（JSON） |
| status | TEXT | 'pending' | 状态 |

#### 状态枚举值
- `pending`: 待执行
- `running`: 执行中
- `completed`: 执行完成
- `failed`: 执行失败

### 需求：Web 页面支持

`test_manager` Web 页面应支持创建和编辑 market_aware 测试计划：

#### 场景：创建测试计划页面
- **当** 用户选择创建 market_aware 类型的测试计划
- **则** 页面显示以下配置区域：
  1. 基本信息：计划名称、描述
  2. 必选参数：标的、时间范围
  3. 振幅参数：decay_factor、stop_loss、total_amount（从配置文件读取默认值）
  4. 行情算法参数：algorithm（默认 always_ranging）
  5. 入场算法参数：可选配置
  6. 首次入场超时参数：a1_timeout_minutes（可选）

#### 场景：参数默认值填充
- **当** 用户打开创建页面
- **则** 各参数子项自动填充默认值，用户可修改

#### 场景：基于已有测试项新建
- **当** 用户点击"基于此测试项新建"
- **则** 自动填充已有测试项的所有参数值

### 需求：测试报告输出

测试报告和详情输出格式保持不变：

#### 场景：输出格式
- **当** 测试完成
- **则** 输出 md 和 csv 文件到现有目录结构（out/test_report/market_aware/）

#### 场景：数据库关联
- **当** 测试完成
- **则** 结果保存到 test_results 表，交易详情保存到 trade_details 表，通过测试 ID 关联

## 移除需求

### 需求：旧 CLI 参数
**原因**: 参数过于分散，难以维护
**迁移方式**: 使用新的 JSON 格式参数替代
