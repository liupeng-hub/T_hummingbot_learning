# 资金递进管理方案 Spec

## 背景

当前回测程序每次都使用固定的 10000 USDT 进行交易，无法利用盈利资金进行复利交易，资金利用率较低。需要设计一种资金递进管理方案，在控制风险的同时提高资金利用率。

## 变更内容

- 新增资金管理模式：固定模式和递进模式
- 新增资金池管理逻辑
- 新增提现（利润锁定）机制
- 新增爆仓恢复机制
- 新增资金管理统计指标

## 影响范围

- 影响代码:
  - `binance_backtest.py` (MarketAwareBacktestEngine 类)
  - `autofish_core.py` (可能需要调整配置类)
  - 数据库表结构（可能需要新增字段记录资金变化）

## 新增需求

### 需求: 双模式资金管理

系统 SHALL 支持两种资金管理模式：

#### 模式 1: 固定模式（现有模式）

- 每次交易使用固定的初始资金（如 10000 USDT）
- 盈利和亏损不影响后续交易资金
- 适合稳定性测试和参数对比

#### 模式 2: 递进模式（新增模式）

- 初始资金为 10000 USDT
- 盈利后资金池增加，可使用更多资金交易
- 亏损后资金池减少，使用更少资金交易
- 爆仓后从利润池中提取资金重新开始

#### 场景: 固定模式运行

- **当** 用户选择固定模式
- **则** 系统每次交易使用固定金额（如 10000 USDT）
- **且** 盈利和亏损不影响后续交易资金

#### 场景: 递进模式运行

- **当** 用户选择递进模式
- **则** 系统根据资金池动态调整交易金额
- **且** 实现利润锁定和爆仓恢复机制

### 需求: 资金池管理

系统 SHALL 实现资金池管理逻辑：

#### 资金池结构

```python
class CapitalPool:
    """资金池管理"""
    initial_capital: Decimal        # 初始资金（如 10000 USDT）
    trading_capital: Decimal        # 当前交易资金（动态变化）
    profit_pool: Decimal            # 利润池（锁定的利润）
    total_profit: Decimal           # 累计利润
    total_loss: Decimal             # 累计亏损
    max_capital: Decimal            # 历史最高资金
    withdrawal_count: int           # 提现次数
    liquidation_count: int          # 爆仓次数
```

#### 资金池更新规则

1. **盈利更新**:
   - 盈利后，trading_capital 增加
   - 如果 trading_capital > initial_capital * 1.5，触发提现检查

2. **亏损更新**:
   - 亏损后，trading_capital 减少
   - 如果 trading_capital < initial_capital * 0.1，触发爆仓检查

3. **提现规则**:
   - 当 trading_capital > initial_capital * 2.0 时
   - 将超出部分转入 profit_pool
   - trading_capital 重置为 initial_capital * 1.5

4. **爆仓恢复**:
   - 当 trading_capital < initial_capital * 0.1 时
   - 如果 profit_pool >= initial_capital:
     - 从 profit_pool 提取 initial_capital
     - trading_capital 重置为 initial_capital
   - 否则:
     - 交易终止，记录最终亏损

#### 场景: 盈利后资金池更新

- **当** 交易盈利 1000 USDT
- **且** 当前 trading_capital 为 10000 USDT
- **则** trading_capital 更新为 11000 USDT
- **且** total_profit 增加 1000 USDT

#### 场景: 提现触发

- **当** trading_capital 达到 20000 USDT（初始资金的 2 倍）
- **则** 将 5000 USDT 转入 profit_pool
- **且** trading_capital 重置为 15000 USDT
- **且** withdrawal_count 增加 1

#### 场景: 爆仓恢复

- **当** trading_capital 降至 1000 USDT（初始资金的 10%）
- **且** profit_pool 有足够资金
- **则** 从 profit_pool 提取 10000 USDT
- **且** trading_capital 重置为 10000 USDT
- **且** liquidation_count 增加 1

#### 场景: 爆仓且无利润池

- **当** trading_capital 降至 1000 USDT
- **且** profit_pool 不足 10000 USDT
- **则** 交易终止
- **且** 记录最终资金和亏损

### 需求: 提现（利润锁定）机制

系统 SHALL 实现利润锁定机制：

#### 提现触发条件

| 条件 | 说明 |
|------|------|
| trading_capital >= initial_capital * 2.0 | 触发提现 |
| 提现金额 = trading_capital - initial_capital * 1.5 | 锁定利润 |
| trading_capital 重置为 initial_capital * 1.5 | 保留安全边际 |

#### 提现策略

**保守策略**（默认）:
- 提现阈值: 2.0 倍初始资金
- 保留资金: 1.5 倍初始资金
- 提现比例: 25%

**激进策略**:
- 提现阈值: 1.5 倍初始资金
- 保留资金: 1.2 倍初始资金
- 提现比例: 20%

**保守策略**:
- 提现阈值: 3.0 倍初始资金
- 保留资金: 2.0 倍初始资金
- 提现比例: 33%

#### 场景: 保守策略提现

- **当** trading_capital 达到 20000 USDT
- **且** 使用保守策略
- **则** 提现 5000 USDT 到 profit_pool
- **且** trading_capital 重置为 15000 USDT

### 需求: 爆仓恢复机制

系统 SHALL 实现爆仓恢复机制：

#### 爆仓判定

- trading_capital < initial_capital * 0.1 (如 1000 USDT)
- 或连续多次大额亏损导致资金不足

#### 恢复流程

```
爆仓触发
    ↓
检查 profit_pool
    ↓
┌─────────────────┐
│ profit_pool >=  │  是  → 从 profit_pool 提取 initial_capital
│ initial_capital │       → trading_capital = initial_capital
└─────────────────┘       → 继续交易
    │ 否
    ↓
交易终止
记录最终亏损
```

#### 场景: 爆仓后恢复成功

- **当** trading_capital 降至 500 USDT
- **且** profit_pool 有 15000 USDT
- **则** 从 profit_pool 提取 10000 USDT
- **且** trading_capital 重置为 10000 USDT
- **且** liquidation_count 增加 1
- **且** 继续交易

#### 场景: 爆仓后无法恢复

- **当** trading_capital 降至 500 USDT
- **且** profit_pool 只有 5000 USDT
- **则** 交易终止
- **且** 记录最终资金: 5500 USDT
- **且** 记录总亏损: 4500 USDT

### 需求: 资金管理统计指标

系统 SHALL 提供详细的资金管理统计：

#### 统计指标

| 指标 | 说明 |
|------|------|
| initial_capital | 初始资金 |
| final_capital | 最终资金 |
| total_return | 总收益率 |
| max_capital | 历史最高资金 |
| max_drawdown | 最大回撤 |
| withdrawal_count | 提现次数 |
| total_withdrawal | 累计提现金额 |
| liquidation_count | 爆仓次数 |
| capital_utilization | 资金利用率 |
| profit_per_trade | 平均每笔盈利 |
| loss_per_trade | 平均每笔亏损 |

#### 场景: 生成资金管理报告

- **当** 回测完成
- **则** 系统生成资金管理统计报告
- **且** 包含所有关键指标
- **且** 可视化资金变化曲线

### 需求: CLI 参数支持

系统 SHALL 通过 CLI 参数支持资金管理模式选择：

```bash
# 固定模式（默认）
python binance_backtest.py \
    --symbol BTCUSDT \
    --date-range 20200101-20260310 \
    --capital-mode fixed \
    --initial-capital 10000

# 递进模式
python binance_backtest.py \
    --symbol BTCUSDT \
    --date-range 20200101-20260310 \
    --capital-mode progressive \
    --initial-capital 10000 \
    --withdrawal-strategy conservative

# 自定义提现策略
python binance_backtest.py \
    --symbol BTCUSDT \
    --date-range 20200101-20260310 \
    --capital-mode progressive \
    --withdrawal-threshold 2.0 \
    --withdrawal-retain 1.5
```

#### CLI 参数说明

| 参数 | 必选 | 说明 |
|------|------|------|
| --capital-mode | 否 | 资金管理模式: fixed/progressive (默认: fixed) |
| --initial-capital | 否 | 初始资金 (默认: 10000) |
| --withdrawal-strategy | 否 | 提现策略: conservative/aggressive/very_conservative |
| --withdrawal-threshold | 否 | 提现阈值倍数 (如 2.0 表示 2 倍) |
| --withdrawal-retain | 否 | 提现后保留倍数 (如 1.5 表示 1.5 倍) |

#### 场景: CLI 参数解析正确

- **当** 用户指定 --capital-mode progressive
- **则** 系统使用递进模式
- **且** 根据其他参数配置资金管理策略

`MarketAwareBacktestEngine` 类 SHALL 新增以下功能：

```python
class MarketAwareBacktestEngine:
    def __init__(
        self,

        entry: Dict = None,

        withdrawal_strategy: str = 'conservative',  # 新增

    def _update_capital_after_trade(self, profit: Decimal):
        """交易后更新资金池"""
        
    def _check_withdrawal(self):
        """检查是否触发提现"""
        
    def _check_liquidation(self):
        """检查是否爆仓"""
        
    def _recover_from_liquidation(self) -> bool:
        """爆仓恢复"""
```

#### 场景: 资金池初始化

- **当** 创建 MarketAwareBacktestEngine 实例

- **则** 系统初始化 CapitalPool

## 风险评估

### 潜在风险

1. **资金计算精度**: Decimal 类型转换和计算
2. **爆仓判定时机**: 需要在合适的时机检查
3. **提现逻辑复杂性**: 多种策略需要充分测试
4. **回测性能**: 频繁的资金池更新可能影响性能

### 缓解措施

1. 使用 Decimal 类型确保精度
2. 在每次交易后检查资金状态
3. 提供详细的单元测试覆盖
4. 优化资金池更新逻辑，减少不必要的计算
