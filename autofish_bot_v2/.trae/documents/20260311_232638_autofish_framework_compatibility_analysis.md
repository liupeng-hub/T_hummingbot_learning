# Autofish 策略与 Freqtrade/Hummingbot 框架兼容性分析

## Autofish 策略核心特点

### 1. 链式挂单机制
- 挂单等待成交（限价单）
- 成交后触发下一层挂单
- 多层级递进（A1 → A2 → A3 → A4）

### 2. 事件驱动架构
- WebSocket 实时监听订单成交事件
- 成交后立即下止盈止损单（条件单）
- 状态机管理：pending → filled → closed

### 3. 订单状态管理
```python
订单状态流转:
pending (挂单中) → filled (已成交) → closed (已平仓)
                  ↓
              下止盈止损单
```

### 4. 核心操作
- 限价入场单（GTC）
- 止盈止损单（条件单，OCO/STOP_MARKET/TAKE_PROFIT_MARKET）
- 订单取消与重建
- 状态持久化与恢复

## Freqtrade 框架分析

### 框架特点
- **信号驱动**：基于技术指标生成买卖信号
- **周期执行**：按固定时间间隔（如每分钟）执行策略
- **订单类型**：主要支持市价单、限价单
- **回测方式**：基于历史 K 线数据模拟

### 兼容性评估

| 特性 | Autofish 需求 | Freqtrade 支持 | 兼容性 |
|------|--------------|---------------|--------|
| 限价挂单 | ✅ 必须 | ✅ 支持 | ✅ 兼容 |
| 止盈止损单 | ✅ 必须（条件单） | ⚠️ 支持（但非实时） | ⚠️ 部分 |
| 订单成交事件 | ✅ 实时监听 | ❌ 轮询检查 | ❌ 不兼容 |
| 链式挂单 | ✅ 成交触发下一单 | ❌ 不支持 | ❌ 不兼容 |
| 状态持久化 | ✅ 必须 | ⚠️ 有限支持 | ⚠️ 部分 |
| 回测支持 | ✅ 需要 | ⚠️ 信号驱动回测 | ❌ 不兼容 |

### 主要问题

1. **事件驱动 vs 轮询驱动**
   - Freqtrade 是周期性轮询执行
   - Autofish 需要实时响应订单成交事件
   - 无法实现"成交后立即下止盈止损单"

2. **回测机制差异**
   - Freqtrade 回测基于信号触发
   - Autofish 回测需要模拟限价单成交逻辑
   - 无法准确模拟"挂单等待成交"的场景

3. **条件单支持有限**
   - Freqtrade 的止盈止损是在持仓后设置
   - Autofish 需要在成交瞬间下条件单
   - 交易所特定的条件单类型支持有限

### 结论：❌ 不适合

Freqtrade 的架构设计是为"信号驱动交易"优化的，不适合 Autofish 的"事件驱动链式挂单"模式。

---

## Hummingbot 框架分析

### 框架特点
- **做市商策略**：专注于流动性提供
- **订单簿驱动**：实时监听订单簿变化
- **纯做市策略**：持续在买卖两侧挂单
- **套利策略**：跨交易所套利

### 兼容性评估

| 特性 | Autofish 需求 | Hummingbot 支持 | 兼容性 |
|------|--------------|-----------------|--------|
| 限价挂单 | ✅ 必须 | ✅ 支持 | ✅ 兼容 |
| 止盈止损单 | ✅ 必须（条件单） | ⚠️ 有限支持 | ⚠️ 部分 |
| 订单成交事件 | ✅ 实时监听 | ✅ 支持 | ✅ 兼容 |
| 链式挂单 | ✅ 成交触发下一单 | ⚠️ 需自定义 | ⚠️ 需开发 |
| 状态持久化 | ✅ 必须 | ✅ 支持 | ✅ 兼容 |
| 回测支持 | ✅ 需要 | ❌ 不支持 | ❌ 不兼容 |

### 主要问题

1. **策略类型不匹配**
   - Hummingbot 主要用于做市商策略
   - Autofish 是趋势跟随/网格混合策略
   - 需要大量自定义开发

2. **无回测功能**
   - Hummingbot 没有内置回测引擎
   - 只能通过 paper trading 测试
   - 无法进行历史数据回测

3. **条件单支持有限**
   - Hummingbot 主要处理限价单
   - 交易所特定的止盈止损单需要额外开发

### 结论：⚠️ 部分适合

Hummingbot 的事件驱动架构与 Autofish 更匹配，但需要大量自定义开发，且缺乏回测功能。

---

## 对比总结

| 维度 | Autofish 原生实现 | Freqtrade | Hummingbot |
|------|------------------|-----------|------------|
| 事件驱动 | ✅ 原生支持 | ❌ 轮询模式 | ✅ 原生支持 |
| 链式挂单 | ✅ 原生支持 | ❌ 不支持 | ⚠️ 需开发 |
| 止盈止损单 | ✅ 原生支持 | ⚠️ 有限 | ⚠️ 有限 |
| 回测支持 | ✅ 自建回测 | ✅ 内置 | ❌ 无 |
| 实盘支持 | ✅ Binance API | ✅ 多交易所 | ✅ 多交易所 |
| 开发成本 | 已完成 | ❌ 无法实现 | ⚠️ 高 |

## 建议

### 1. 保持现有架构
Autofish 策略的核心特性（事件驱动、链式挂单、条件单）与 Freqtrade/Hummingbot 的设计理念存在本质差异，建议保持现有的独立实现。

### 2. 可借鉴的部分

**从 Freqtrade 借鉴**：
- 策略参数化配置方式
- 回测报告生成格式
- 风险管理模块

**从 Hummingbot 借鉴**：
- 多交易所适配器设计
- 订单簿数据处理
- 日志和监控架构

### 3. 如果必须迁移

**迁移到 Hummingbot 的方案**：
1. 继承 `ScriptStrategyBase` 类
2. 重写 `on_tick()` 方法实现链式挂单逻辑
3. 使用 `c_execute_proposal()` 执行订单
4. 自定义止盈止损单处理
5. 无回测，只能 paper trading

**迁移成本**：高，且功能受限

### 4. 最终结论

**不推荐迁移到 Freqtrade 或 Hummingbot**

原因：
1. Freqtrade 的轮询架构无法实现事件驱动的链式挂单
2. Hummingbot 缺乏回测功能，且需要大量自定义开发
3. 现有 Autofish 实现已经完整覆盖核心功能
4. 迁移成本高，收益低

## 其他框架分析

### Backtrader

**框架特点**:
- **事件驱动**: `notify_order()` 回调函数处理订单状态变化
- **OCO 订单**: 支持 One-Cancels-Other 订单组合
- **Bracket Order**: 支持止损止盈组合单
- **回测引擎**: 内置完整的回测系统

**兼容性评估**:

| 特性 | Autofish 需求 | Backtrader 支持 | 兼容性 |
|------|--------------|-----------------|--------|
| 限价挂单 | ✅ 必须 | ✅ 支持 | ✅ 兼容 |
| 止盈止损单 | ✅ 必须 | ✅ Bracket Order | ✅ 兼容 |
| 订单成交事件 | ✅ 实时监听 | ✅ notify_order | ✅ 兼容 |
| 链式挂单 | ✅ 成交触发下一单 | ⚠️ 需自定义 | ⚠️ 需开发 |
| 回测支持 | ✅ 需要 | ✅ 内置 | ✅ 兼容 |

**示例代码**:
```python
def notify_order(self, order):
    if order.status == order.Completed:
        # 订单成交后触发下一层挂单
        if order.isbuy():
            self.buy(size=next_level_size, price=next_price)
            # 设置止损止盈
            self.sell(exectype=Order.Stop, price=stop_price)
            self.sell(exectype=Order.Limit, price=take_profit_price)
```

**结论**: ⚠️ 部分适合
- 事件驱动架构匹配
- 支持 OCO/Bracket Order
- 链式挂单需要自定义实现
- 回测功能完善

### VNPY (VeighNa)

**框架特点**:
- **事件驱动**: 中央事件引擎 (Event Engine) 解耦各模块
- **StopOrder**: 本地停止单，条件触发单
- **多交易所**: 支持国内外主流交易所
- **回测引擎**: 内置回测模块

**兼容性评估**:

| 特性 | Autofish 需求 | VNPY 支持 | 兼容性 |
|------|--------------|-----------|--------|
| 限价挂单 | ✅ 必须 | ✅ 支持 | ✅ 兼容 |
| 止盈止损单 | ✅ 必须 | ✅ StopOrder | ✅ 兼容 |
| 订单成交事件 | ✅ 实时监听 | ✅ EVENT_ORDER | ✅ 兼容 |
| 链式挂单 | ✅ 成交触发下一单 | ⚠️ 需自定义 | ⚠️ 需开发 |
| 回测支持 | ✅ 需要 | ✅ 内置 | ✅ 兼容 |

**示例代码**:
```python
def on_order(self, order):
    if order.status == Status.ALLTRADED:
        # 订单全部成交后触发下一层
        new_order = self.buy(price=next_price, volume=next_volume)
        # 设置止损止盈
        self.sell(price=stop_price, stop=True)
        self.sell(price=take_profit_price, stop=True)
```

**结论**: ⚠️ 部分适合
- 事件驱动架构完全匹配
- 支持条件单
- 链式挂单需要自定义实现
- 国内交易所支持更好

- **订单类型**: 限价单、市价单、止损单

**兼容性评估**:

| 特性 | Autofish 需求 | PyAlgoTrade 支持 | 兼容性 |
|------|--------------|------------------|--------|
| 限价挂单 | ✅ 必须 | ✅ 支持 | ✅ 兼容 |
| 止盈止损单 | ✅ 必须 | ⚠️ 有限 | ⚠️ 部分 |
| 订单成交事件 | ✅ 实时监听 | ✅ onOrderUpdated | ✅ 兼容 |
| 链式挂单 | ✅ 成交触发下一单 | ⚠️ 需自定义 | ⚠️ 需开发 |
| 回测支持 | ✅ 需要 | ✅ 内置 | ✅ 兼容 |

**结论**: ⚠️ 部分适合
- 事件驱动架构
- OCO 订单支持有限
- 主要支持股票/加密货币

## 新建议

### 如果必须选择框架迁移

**推荐优先级**: VNPY > Backtrader > Hummingbot > PyAlgoTrade > Freqtrade

**VNPY 优势**:
1. 事件驱动架构与 Autofish 完全匹配
2. 内置回测引擎
3. 支持多交易所（包括 Binance）
4. 国内社区活跃，文档丰富
5. StopOrder 条件单支持

**迁移到 VNPY 的方案**:
1. 继承 `CtaTemplate` 类
2. 在 `on_order()` 中实现链式挂单逻辑
3. 使用 `self.buy()` / `self.sell()` 下单
4. 使用 `stop=True` 参数设置条件单
5. 回测使用 `BacktestingEngine`

**迁移成本**: 中等（需要重写策略逻辑，但框架支持良好）

### 最终建议

**仍然推荐保持现有独立实现**

原因：
1. 现有 Autofish 实现已经完整覆盖核心功能
2. 迁移需要重写策略逻辑，成本不低
3. 框架的通用性可能限制特定优化

**如果需要框架的优势**:
- **需要多交易所支持**: 考虑 VNPY
- **需要更完善的回测**: 考虑 Backtrader
- **需要社区生态**: 考虑 VNPY（国内）或 Freqtrade（国外）
